# 生意参谋竞品分析 Dashboard

## 目标与边界

将生意参谋竞争模块的 Excel 转换为可审计、可交互的竞品分析 Dashboard。优先保证口径正确和可追溯，再考虑图表丰富度。

不要把参考脚本当成通用成品。Sheet 名称、字段顺序、导出窗口和用户自建辅助表可能变化，必须按实际文件适配。

不要默认执行以下决定：排除本店、竞品映射、价格带、近似日期窗口合并、参考价格和分析模块删减。

## 必读资源

- 首次处理数据或修改解析逻辑前，读 `references/data-spec.md`。
- 构建新 Dashboard 或修改前端结构时，读 `references/dashboard-architecture.md`。
- 每次修改现有 Dashboard 或新增竞品前，读 `references/pitfalls.md`。
- 编写提取逻辑时参考 `scripts/extract_template.py`；按实际表头映射，不要原样复制。
- 输出 JSON 后运行 `scripts/validate_data.py`；验证失败时不要继续构建 Dashboard。

## 强制工作流

### 1. 只读审计输入

通读所有 Sheet，不跳读。先输出一份数据审计摘要，至少包含：

- 文件、Sheet、行列数和真实表头
- 数据覆盖的店铺、商品 ID、类别和日期窗口
- 数值样例、无法解析值、空值、重复业务键和异常字段
- 同名店铺、同商品多链接、不同文件的类别编号冲突
- 标准自然月与“近 30 天”窗口的差异

读取公式结果时使用 `data_only=True`；如关键公式没有缓存值，停止并说明，不能把空值当零。

### 2. 建立业务口径门禁

在生成最终 Dashboard 前，与用户确认：

1. 分析对象以及需排除的本店/品牌
2. 竞品名称、商品链接映射、参考价格和价格带
3. 日期窗口如何比较；非同一窗口是否分开展示或经用户同意后映射
4. 新建还是兼容现有 Dashboard，以及需要保留的模块

用户未提供参考价格时，不生成 GMV 数字；可以保留模块占位或暂不展示。

### 3. 规范化并验证数据

- 将区间值解析为 `{low, mid, high}`；精确值增加 `exact: true`。
- 将缺失值保存为 `null`，不要把 `-`、空白或无法解析值改成零。
- 对非空但无法解析的值报错，并记录 Sheet、行号、字段和原值。
- 清洗商品 ID，但保留原始值用于审计；商品链接只使用纯数字 ID。
- 用真实日期键建立全局日期并集，再逐对象对齐；不要复制第一个对象的日期轴。
- 不自动合并日期相近的“近 30 天”窗口。只有用户确认业务等价后才映射，并在模块说明中披露。
- 以统一观察窗口内的累计买家中值选择 Top1 链接；窗口不一致时不可直接排名。
- 保留生意参谋给出的转化率字段。若另行计算 `买家数 / 访客数`，必须使用不同名称并说明其与官方区间可能不一致。

先输出独立 JSON，运行：

```bash
python scripts/validate_data.py output.json
```

验证通过后再嵌入 HTML。

### 4. 构建 Dashboard

默认模块：

- 店铺：销售趋势、流量结构、关键词、竞争格局
- 单品：主力链接趋势、GMV 估算、流量下钻、关键词、SKU、策略推断

单品趋势、GMV、流量、关键词和策略默认使用 Top1 链接；SKU 默认汇总所有有效链接，并清楚标注覆盖链接数。

每个模块都显示数据来源、时间窗口、估值口径和计算方式。策略推断必须展示证据，不把相关性写成因果结论。

GMV 只能表述为“参考价 × 支付买家数”的估算。参考价不是实际成交均价；不要使用“真实成交额”“实际销售额”等表述。

使用固定版本的 Chart.js CDN 构建单 HTML。店铺与单品流量分别遵守既定数据结构，不为统一形式增加迁移层。

前端结构与安全嵌入方式见 `references/dashboard-architecture.md`。

### 5. 验证交付物

每次修改后完成以下检查：

1. 数据验证脚本通过，无残留数值字符串、非法区间和字段缺失
2. 提取后的自有 JavaScript 通过 `node --check`
3. 所有渲染目标存在，事件引用均有定义
4. Chart.js 实例重绘前已销毁
5. 低/中/高估值切换刷新全部受影响模块
6. Tab、筛选、月份切换、流量展开/收起和商品跳转可用
7. 浏览器无控制台错误
8. 输出文件名包含版本号，保留上一版用于回溯

修改现有 Dashboard 时，额外执行 `references/pitfalls.md` 的回归清单。

## 协作输出

在真正构建前，先向用户提交：

- 数据审计摘要
- 待确认的业务口径
- 拟保留模块与分组
- 已知限制和不会做的推断

完成后交付 Dashboard、数据口径摘要和验证结果。不要用大量实现细节替代业务结论。
